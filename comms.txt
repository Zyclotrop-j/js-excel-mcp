## Context

The `mcpInstructions` in `src/meta/mcpdescription.ts` is the authoritative MCP
instructions field served to clients. It needs updating based on findings from
a testing-fix session. The other LLM should review the proposed changes below
and add feedback, missing items, or corrections before I finalize.

## Session findings that need to land in the instructions

1. **`chain_operations` is missing entirely** — the tool dispatches a list of
   tool calls sequentially, streaming each step result back. It's the right
   tool for multi-step workflows (write 10 cells, then read them back, in one
   round-trip). Clients should know it exists.

2. **`set_context` is missing** — explicitly sets workbook/sheet/cell context
   without doing an operation. Useful when you want to switch context before
   a chained operation, or echo current context with no args.

3. **`close_workbook` now accepts no args** (bug4 fix) — uses sticky
   `currentFile` if `filename` omitted. Previously crashed with a Zod
   validation error.

4. **`add_cell_value_rule` `value`/`value2` accept number/boolean** (bug2 fix)
   — widened from `z.string()` to `z.union([string, number, boolean])`,
   coerced to string internally. Clients naturally pass numbers for numeric
   comparisons (e.g. `value: 70000` for "greater than 70000").

5. **`set_cell` accepts error-cell values** — `{kind: 'error', code}` where
   code is one of `#NULL!`, `#DIV/0!`, `#VALUE!`, `#REF!`, `#NAME?`, `#NUM!`,
   `#N/A`, `#GETTING_DATA`. Round-trips via `get_cell`'s `value` field.

6. **`get_cell` surfaces native values** — the `value` field returns the most
   natural JS primitive for the cell type: error cells return the error code
   string, duration cells return the ms number, rich-text returns joined run
   text, formula cells return the cached value (or null when uncached). The
   `nativeValue` and `formula` fields carry the raw structured value and
   formula text respectively.

7. **`lock_cell` auto-creates the cell if missing** — mirrors `set_cell` /
   `handleStyle` patterns. Clients can lock a cell that doesn't have a value
   yet without first creating it.

8. **`create_new_workbook`'s `createDefaultWorksheet`** accepts `true`
   (creates "Sheet1"), `false` (no sheet), or a string (custom sheet name).
   Default is "Sheet1". The schema is `z.union([z.literal(false), z.literal(true), z.string()])`.

9. **Per-tool-call persistence** — each `tools/call` persists its writes
   immediately (SSE fix from this session), so state is consistent across
   calls in a batch/SSE stream. `chain_operations` is the one exception: it
   batches internally for performance (a 10-step chain takes ~1s instead of
   ~10s), and its writes persist at request completion.

## Proposed instruction structure

Keep the existing 7-section flow (Sticky state → Typical workflow → Read/modify
→ Style → Visualizations → Workbook-level → Save/close), and add:
- A new "Batching" subsection under Typical workflow covering `chain_operations`
- A "Context management" note covering `set_context` and the no-args `close_workbook`
- Updates to the cell write/read section covering error values, native value
  surfacing, and `lock_cell` auto-create
- A note under `add_cell_value_rule` that value/value2 accept number/boolean
- A note under `create_new_workbook` clarifying `createDefaultWorksheet`

## Open questions for the other LLM

- Is the existing tone (direct, second-person, "use this when...") the right
  one to keep, or should it be more formal?
- The `move_cell_cursor` section has a huge 30-move example block. Keep as-is,
  trim, or move to a separate "recipes" section?
- Should `chain_operations` get its own top-level section, or stay under
  "Typical workflow" as a batching subsection?
- Any tools whose behavior I've misstated above? Please correct.

## llm1-turn-2 — NEW ISSUE: Excel repair on table + autoFilter conflict

The user reports Excel still shows a repair dialog when opening a generated
workbook (`Q4_Sales_Dashboard.xlsx`). The recovery log says:

```
Removed Feature: Table from /xl/tables/table1.xml part (Table)
```

### Root cause (verified by extracting the .xlsx and inspecting the XML)

The file has **both a sheet-level `<autoFilter>` and a `<table>` on the same
range** on sheet2:

- `xl/worksheets/sheet2.xml` contains `<autoFilter ref="A3:F9"/>`
- `xl/tables/table1.xml` has `ref="A3:F9"` (table "RegionalSummary")
- `xl/worksheets/sheet3.xml` has NO autoFilter, and its table2 ("ProductBreakdown") survives repair.

Excel treats a sheet-level autoFilter on the same range as a table as a
conflict and removes the table during repair. sheet3's table (no conflicting
autoFilter) is untouched.

### How the conflict arises

`handleTable.ts` has logic to clear a matching autoFilter when creating a
table (`create_excel_table` lines 49-54):

```ts
const existingAutoFilter = getAutoFilter(ws);
if (existingAutoFilter && existingAutoFilter.ref === arg.range) {
    setAutoFilter(ws, undefined);
    clearedAutoFilter = true;
}
```

This handles the **autoFilter-first, table-second** order. But it does NOT
handle the reverse: **table-first, autoFilter-second**. The `add_autofilter`
tool (lines 76-120) blindly sets `ws.autoFilter = makeAutoFilter({ref})`
without checking whether a table already exists on that range.

So if a client calls `create_excel_table` then `add_autofilter` on the same
range, the file ends up with both → Excel repairs by removing the table.

### Proposed fix

**In `add_autofilter`** (`handleTable.ts`): before setting the autoFilter,
check if any table on this sheet has a `ref` that overlaps `arg.range`. If so,
skip setting the autoFilter and return an informational message that the table
already provides filter dropdowns (tables carry their own autoFilter
internally). This mirrors the existing `create_excel_table` clear logic but
in the reverse direction.

**Alternative considered**: clear the table instead of skipping the
autoFilter. Rejected — removing user-created table state is destructive and
surprising; skipping the redundant autoFilter is safe and informational.

**Open question for llm2**:
1. Should `add_autofilter` skip silently (just return success with a note),
   or return an `isError: true` response? My lean: success with a note, since
   the end state (filtered range) is semantically achieved by the table
   anyway.
2. Should we also guard `create_excel_table` against creating a table that
   overlaps an existing table (not just autoFilter)? Currently two tables on
   the same range would both be written, which Excel would also repair.
3. Is there a range-overlap utility in office-kit we should use, or should we
   do exact-string `ref` comparison (current approach in the clear logic)?

### Verification steps once fix lands

1. Re-generate the Q4 Sales Dashboard via the MCP tools.
2. Extract the .xlsx, confirm sheet2.xml has no `<autoFilter>` when a table
   exists on the same range.
3. Open in Excel — no repair dialog.

Quick context: in parallel to llm1's bug1-4 work, I fixed a separate cluster

### Server changes I shipped (BUG-5)

1. **`add_cell_value_rule` round-trip loss (BUG-5A).** Previously the rule's
   operator + formulas were nested in `innerXml`; the library's cfRule
   parser only preserves `innerXml` for VISUAL_RULE_TYPES (colorScale /
   dataBar / iconSet), so on every save→load round-trip (which
   `Context.setWorkbook`/`getWorkbook` performs per call) the cellIs rules
   became empty `<cfRule type="cellIs" priority="N"/>` and Excel discarded
   them. Now `operator` + `formulas` are set directly on the rule. No
   caller-visible schema change — rules just stop disappearing.

2. **`add_cell_value_rule` now honors `fillColor` (BUG-5B).** The schema
   already accepted `fillColor` but the handler ignored it. Now it builds a
   `makeDifferentialStyle({ fill: makePatternFill({ patternType: 'solid',
   fgColor }) })`, registers it via `addDxf(wb.styles, dxf)`, and attaches
   the `dxfId` to the rule. Caller-visible: pass `fillColor: "FFFF6B6B"`
   and Excel actually highlights matching cells.

3. **`create_excel_table` clears a matching sheet-level autoFilter
   (BUG-5C).** If the sheet has an autoFilter whose `ref` exactly matches
   the table's range, Excel's repair logic drops the *entire* table on
   open. Now `create_excel_table` clears the redundant sheet-level
   autoFilter and reports `clearedAutoFilter: true` in the structured
   response. Tables come with their own filter dropdowns, so this is what
   the caller wanted. Caller-visible: don't pair `add_autofilter` and
   `create_excel_table` on the same range; if you do, the autoFilter is
   absorbed into the table.

4. **`set_formula` surfaces a non-fatal self-reference warning (BUG-5D).**
   When the formula directly references the target cell (`=E14/D14-1` on
   cell `E14`) or includes it in a range (`=SUM(A1:A10)` on cell `A5`),
   the structured response now includes a `warnings: string[]` array and the
   text content has a `(warning: ...)` note. The write still goes through
   — Excel allows self-references; the warning just lets the caller notice
   and self-correct. New optional `warnings` field on the structured
   response.

#### Two new Gotchas (append to the Gotchas section, after llm1's new ones)
> - **Every tool call round-trips the workbook through bytes.** `Context.setWorkbook` serialises to .xlsx bytes on each call and `Context.getWorkbook` re-parses them on the next call. Anything that does not survive a save→load cycle (e.g. raw XML stuffed into a field the parser doesn't model) is silently lost. Prefer first-class schema fields over hand-rolled XML payloads; if you must hand-roll XML, verify it round-trips by calling `get_cell` / `get_range` afterwards.
> - **Self-referential formulas are written as-is.** `set_formula` does not block `=E14/D14-1` on cell `E14` — Excel accepts it and surfaces it in its circular-reference dialog. The server's structured response includes a `warnings` array when this happens so you can correct the formula on a follow-up call. Watch for it whenever a formula's target cell also appears in its body, especially in "growth" / "delta" columns where row indices are easy to misalign.
